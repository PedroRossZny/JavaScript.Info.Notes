# JavaScript Symbol Type

## Exercise 133

Only two primitive types can serve as object property keys: the string type and the symbol
type. If another type is used, such as a number, it will be converted to a string.

A *symbol* represents a unique identifier. A value of this type can be created using
`Symbol()`. When creating a symbol, we can also assign a description, also referred to as
the symbol's name. If we create two symbols with the same description, they will still
have different values — the description is just a label and has no effect on anything.

In summary, a symbol is a *unique primitive value* with an optional description.

In JavaScript, most values support implicit conversion to string, but symbols are special
— strings and symbols are fundamentally different and should not be converted into each
other. If we want that to happen, we use `.toString()` to explicitly convert a symbol to
a string, or the `symbol.description` property to show only the symbol's description, as
shown in **ex133**.

## Exercise 134

Symbols allow us to create hidden properties of an object. For instance, if we are working
with an object that belongs to a third-party codebase and we want to add identifiers to
it, we can use symbols for that.

The advantage of using symbols instead of strings is that since the object belongs to
another codebase, it is not safe to add string fields to it, as that could affect its
behavior. Newly defined symbols however are safe to add, since the third-party code will
have no knowledge of them. This also applies when another script wants its own identifier
inside the same object — there will be no conflict between symbols since they are always
unique, even if they share the same description. If we used a string for the same purpose,
there would be a conflict with one value overwriting the other, as shown in **ex134**.

## Exercise 136

Symbols do not participate in `for..in` loops, and `Object.keys(user)` also ignores them.
This is the core behavior behind hiding symbolic properties — if another script or library
iterates over our object, it will not unexpectedly access a symbolic property.

There is one exception however: `Object.assign` copies both string and symbol properties,
since when cloning an object we generally want all properties to be copied, including
symbols, as shown in **ex136**.

## Exercise 137

Generally all symbols are different even if they share the same description, but sometimes
we need symbols with the same description to represent the same entity. For that we use
global symbols, which guarantee that repeated lookups by the same description always
return the exact same symbol.

To do this we use `Symbol.for(key)`, which checks the global registry. If a symbol
described as `key` exists, it returns it — otherwise it creates a new `Symbol(key)`,
stores it in the registry, and returns it as a global symbol. This is useful when we need
a symbol that is valid across the entire application and accessible from anywhere in the
code. This is similar to the Ruby language, where there is a single symbol for each name
— in JavaScript the same is true for global symbols, as shown in **ex137**.

## Exercise 138

Using `Symbol.for(key)` we can retrieve a global symbol by its description. To do the
opposite — retrieve a description by its global symbol — we use `Symbol.keyFor(sym)`. If
the symbol is not global, it will not be found and `undefined` will be returned. That
said, all symbols have the `description` property regardless, as shown in **ex138**.

## Additional

There are many built-in system symbols that JavaScript uses internally, such as
`Symbol.hasInstance`, `Symbol.isConcatSpreadable`, `Symbol.iterator`,
`Symbol.toPrimitive`, and so on. For example, `Symbol.toPrimitive` allows us to describe
the conversion of objects into primitive types.
