ignores it. However, we generally try to avoid writing numbers with a large quantity of

## Exercise 148

The method `num.toString(base)` returns a string representation of `num` in the numeral
system with the given `base`. The base can range from 2 to 36, with 10 as the default.
The most common use cases are:

- **base 16** — used for hex colors, character encodings, etc. Digits can be `0..9` or
`A..F`.
- **base 2** — mainly used for debugging bitwise operations. Digits can be `0` or `1`.
- **base 36** — the maximum. Digits can be `0..9` or `A..Z`, using the entire Latin
alphabet to represent a number. This base is commonly used to shorten a long numeric
identifier, such as creating a short URL.

If we want to call a method directly on a number, we need to place two dots after it
instead of one. With only one dot, JavaScript assumes a decimal part follows and throws an
error. With two dots, it recognizes that the decimal part is empty and proceeds normally.
We can also wrap the number in parentheses instead, which works the same way, as shown in
**ex148**.

## Exercise 149

Rounding is one of the most common operations when working with numbers. There are several
functions available for this:

- `Math.floor` — rounds down.
- `Math.ceil` — rounds up.
- `Math.round` — rounds to the nearest integer.
- `Math.trunc` *(not supported by Internet Explorer)* — removes everything after the
decimal point without rounding.

Here is a summary of their differences:

| | `3.1` | `3.5` | `3.6` | `-1.1` | `-1.5` | `-1.6` |
|---|---|---|---|---|---|---|
| `Math.floor` | 3 | 3 | 3 | -2 | -2 | -2 |
| `Math.ceil` | 4 | 4 | 4 | -1 | -1 | -1 |
| `Math.round` | 3 | 4 | 4 | -1 | -1 | -2 |
| `Math.trunc` | 3 | 3 | 3 | -1 | -1 | -1 |

We can also round a number to a specific decimal place, such as rounding `1.2345` to
`1.23`. There are two ways to do this:

- **Multiply and divide** — for example, to round to two decimal places, multiply the
number by `100`, apply the rounding function, then divide the result back by `100`.
- **`toFixed(n)`** — rounds the number to `n` digits after the decimal point and returns
a string representation of the result. Its rounding behavior is similar to `Math.round`.
If the decimal part is shorter than needed, zeros are added at the end. To convert the
result back to a number, use `Number()` or the unary addition operator `+num.toFixed(5)`,
as shown in **ex149**.

## Exercise 150

Internally, a number is represented in the 64-bit IEEE-754 format, so there are exactly
64 bits to store a number: 52 are used to store the digits, 11 store the position of the
decimal point, and 1 bit is used for the sign. If a number is too large, it can exceed
the 64-bit limit and become the special numeric value `Infinity`.

Loss of precision also occurs frequently. For example, a simple sum of `0.1` and `0.2`
gives `0.30000000000000004` instead of `0.3`. This happens because numbers are stored in
memory in binary form, and fractions like `0.1` or `0.2` that seem simple in the decimal
system are actually infinite in binary. A useful comparison: division by powers of `10`
works correctly in the decimal system, but division by `3` does not — the same happens in
binary, where division by powers of `2` works fine but `1/10` results in an infinite
fraction.

We can work around this by rounding numbers using `toFixed(n)`, which always returns a
string, but we can convert it back using the unary addition operator. Another approach is
to temporarily multiply the numbers to turn them into integers, perform the calculations,
and then divide back — this reduces the error slightly, but it can still occur during
division.

Another situation where precision is lost is when a number exceeds the 52 bits available
for storing digits, causing the least significant digits to disappear. JavaScript does its
best to fit the number into the desired format, even when the format is not large enough.

One more curiosity: there are two zeros, `0` and `-0`, since the sign is represented by a
single bit. In most cases the distinction is imperceptible, as operators treat them as
equal, as shown in **ex150**.

## Exercise 151

The special values `Infinity`, `-Infinity`, and `NaN` belong to the `number` type, but
since they are not regular numbers, there are special functions to check for them:

- `isNaN(value)` — converts its argument to a number and then checks if it is `NaN`. This
function is necessary because `NaN` is unique in that it is not equal to anything,
including itself.
- `isFinite(value)` — converts its argument to a number and returns `true` if it is a
regular number, and `false` for `NaN`, `Infinity`, or `-Infinity`. It is sometimes used
to validate whether a string value is a regular number. Note that an empty string or a
string containing only spaces is treated as `0`.

The methods `Number.isNaN` and `Number.isFinite` are stricter versions — they do not
automatically convert the argument to a number, but first check whether it belongs to the
`number` type.

There is also a special built-in method `Object.is`, which compares values like `===` but
is more reliable in two cases:

- It works with `NaN`: `Object.is(NaN, NaN) === true`.
- It distinguishes `0` and `-0`: `Object.is(0, -0) === false`. This is technically
correct because internally the number has a sign bit that can differ even when all other
bits are zero.

In all other cases, `Object.is` behaves like `===`, as shown in **ex151**.

## Exercise 152

Numeric conversion using the unary addition operator is strict — if the value is not
exactly a number, the conversion will fail, with the exception of spaces which are
ignored. To convert strings that contain numbers followed by extra characters, we use
`parseInt` for integers and `parseFloat` for floating point numbers. Both read a number
from a string until it is no longer possible to do so. `parseInt` also has an optional
second argument to specify the numeral system base, allowing it to parse hexadecimal,
binary, and other number formats, as shown in **ex152**.
