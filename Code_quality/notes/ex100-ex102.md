# JavaScript Ninja Code

In this section, we will go through practices that should be avoided in your code in order to reinforce good coding habits.

## Exercise 100

Do not try to make your code as short as possible.

Your goal should be to make it as simple and readable as possible, not as short as possible.

An example of what **not** to do is shown in **ex100**.

---

## One-letter Variables

Never use a single letter as a variable name.

Doing so makes variables harder to find and nearly impossible to understand at a glance.

There are a few common exceptions. For example, in a `for` loop, the variable `i` is widely used as a loop counter and is immediately recognizable to most developers.

If you use a single-letter counter variable, stick to common conventions such as `i`, `j`, or `k`.

Do not try to be different by using names such as `x` or `y` for loop counters.

Also, avoid making loop bodies excessively long. If the body spans many lines, it becomes much harder to remember that a variable such as `x` is simply a loop counter.

---

## Abbreviations

Never abbreviate descriptive variable names.

Other developers will not know exactly what the variable stores or what it is used for.

Use complete and meaningful names whenever possible.

---

## Don't Be Abstract

When choosing names, be as specific as possible.

Avoid generic names such as:

* `obj`
* `data`
* `value`
* `item`
* `elem`

Never name variables based solely on their type, such as:

* `str`
* `num`

Likewise, never create a series of similarly named variables by adding numbers:

* `data1`
* `item2`
* `elem5`

These names provide little or no useful information.

---

## Don't Do Attention Tests

Never use variable names that are extremely similar to one another.

For example:

* `date`
* `data`

Names like these make code much slower and more exhausting to read because developers must constantly double-check which variable is being referenced.

---

## Synonyms

Do not randomly switch between synonymous function prefixes.

Choose a naming convention and use it consistently.

For example, if a function named `displayMessage` exists, avoid creating another function named `showName` in the same codebase.

Instead, follow a consistent style, such as:

```javascript
showMessage(...)
showName(...)
```

or:

```javascript
displayMessage(...)
displayName(...)
```

In a team environment, naming conventions should be agreed upon so that everyone follows the same standard.

The opposite is also true: do not use the same prefix for functions that perform fundamentally different actions.

## Exercise 101

Do not reuse the same variable to store unrelated values.

Whenever possible, create new variables.

Never secretly replace the value of a variable with something completely different in the middle of a loop or function.

This makes code difficult to understand and maintain.

As shown in **ex101**.

---

## Underscores

Avoid placing unnecessary underscores around variable names, such as:

```javascript
_name_
__value_
```

This only makes code longer and less readable.

Likewise, do not assign secret meanings to underscores and expect other developers to figure them out.

Code should be explicit rather than cryptic.

---

## Don't Overdo It with Names

Do not invent fancy or exaggerated variable names such as:

```javascript
superElement
megaFrame
niceTime
```

Names like these provide little useful information.

Such names may be acceptable in experiments, jokes, or personal tests, but they should not appear in serious code.

---

## Exercise 102

Do not use the same name for variables inside and outside a function.

A local variable will shadow a global variable with the same name, which can create confusion.

This results in two variables with identical names existing in the same codebase while holding different values and serving different purposes.

As shown in **ex102**.

## Overloading Functions

Many functions such as:

```javascript
isReady()
checkPermission()
findTags()
```

perform calculations, searches, or validations and simply return data without modifying anything outside the function.

Functions like these should focus exclusively on their primary responsibility.

Do not add unrelated actions to them.

For example, a function named `checkPermission()` that returns `true` or `false` should not suddenly start displaying messages, modifying the page, or returning complex objects unrelated to its original purpose.

Keep each function focused on a single task.

---

## Functions Beyond Their Limits

Do not overload functions with additional responsibilities.

A function should do exactly what its name suggests and nothing more.

For example:

```javascript
validateEmail(email)
```

should only validate an email address.

It should not:

* Display error messages.
* Ask the user to enter another email.
* Modify interface elements.
* Perform unrelated business logic.

Those responsibilities belong in separate functions specifically designed for those tasks.

Keeping functions focused makes code easier to understand, test, reuse, and maintain.
