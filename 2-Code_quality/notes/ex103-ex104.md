# JavaScript Automated testing with Mocha

## Exercise 103

During development, we verify that a function works by executing it and comparing the result with the expected output. If something goes wrong, we fix the code and run the function again, repeating this process until everything works as intended. However, when testing code through repeated manual executions, it is easy to overlook something.

Automated testing means that tests are written separately from the code itself. They execute our functions in different ways and compare the results with the expected ones.

To perform automated tests, we will learn about *Behavior-Driven Development* (*BDD*). It combines three things into one: tests, documentation, and examples.

Before writing the implementation of our function, we first imagine what the function should do and describe its behavior. This description is called a *specification* (or *spec*), which contains use case descriptions along with tests.

A specification has three fundamental elements:

- `describe("title", function() { ... })` — indicates which feature is being described and groups related `it` blocks.
- `it("use case description", function() { ... })` — the title describes a specific use case in a human-readable way, while the function contains the test code.
- `assert.equal(value1, value2)` — code inside the `it` block. If the implementation is correct, it runs without errors. The `assert.equal` function checks whether the two values are equal and throws an error if they are different.

The development workflow follows these steps:

1. Create an initial specification covering the basic functionality.
2. Write the first implementation.
3. Run the *Mocha* testing framework. As long as the implementation is incomplete, errors will be displayed. Fix the issues until all tests pass.
4. You now have an initial implementation with passing tests.
5. Add more use cases, which will likely expose unsupported behavior and produce new errors.
6. Repeat step 3 until no errors remain.
7. Continue repeating the process until the feature is complete.

For this development approach, we will use a few libraries:

- *Mocha* — the core testing framework. It provides common testing functions such as `describe` and `it`, as well as the main function that executes the tests.
- *Chai* — an assertion library that provides a wide variety of assertions, including `assert.equal`.
- *Sinon* — a library used to spy on functions, mock native functions, and much more.

All of these libraries work both in browsers and on servers, but in our case we will use them in the browser.

We can divide our HTML page into five parts:

1. Inside the `<head>`, add the third-party testing libraries and styles:
   - `<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/mocha/3.2.0/mocha.css">` — Mocha CSS used to display test results.
   - `<script src="https://cdnjs.cloudflare.com/ajax/libs/mocha/3.2.0/mocha.js"></script>` — the Mocha framework.
   - `<script> mocha.setup('bdd'); </script>` — minimal setup.
   - `<script src="https://cdnjs.cloudflare.com/ajax/libs/chai/3.5.0/chai.js"></script>` — adds Chai.
   - `<script> let assert = chai.assert; </script>` — since Chai provides many features, we make `assert` globally available.

2. Inside the `<body>`, place the `<script>` containing the function we are going to implement and test.

3. After that, include another external `<script>` containing the `describe` code.

4. Add an HTML element `<div id="mocha">`, which Mocha uses to display the test results.

5. Finally, start the tests by calling `mocha.run()` inside a `<script>`.

We will almost always need to test multiple use cases, and there are two common ways to organize them:

1. Add multiple `assert` statements inside the same `it` block.
2. Create multiple `it` blocks, one for each test case.

The second approach is generally recommended because when an error occurs inside an `it` block, execution stops immediately. If the first assertion fails, the remaining assertions are never executed. Running tests separately provides much more information about what is happening.

There is also a useful trick when dealing with a large number of similar tests. Instead of writing every `it` block manually, we can generate them inside a `for` loop.

We can also nest our descriptions by creating a main `describe` block that groups all tests, with nested `describe` blocks for more specific test categories.

When testing mathematical functions, we will often encounter `NaN`, which is the value JavaScript returns for invalid mathematical operations. In those situations, we should use assertions such as `assert.isNaN();` to verify the result.

## Additional

Other useful *assertions* include:

- `assert.isNaN` — verifies that a value is `NaN`.
- `assert.equal(value1, value2)` — checks loose equality using `value1 == value2`.
- `assert.strictEqual(value1, value2)` — checks strict equality using `value1 === value2`.
- `assert.notEqual` / `assert.notStrictEqual` — the opposite of the previous equality assertions.
- `assert.isTrue(value)` — verifies that `value === true`.
- `assert.isFalse(value)` — verifies that `value === false`.

You can find the complete list in the [documentation](https://www.chaijs.com/api/assert/).
