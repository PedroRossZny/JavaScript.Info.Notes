# JavaScript Functions

## Exercise 066

In many programs, we need to perform similar actions multiple times. This is where functions become useful, as they are one of the main *building blocks* of a program.

We have already seen several built-in functions such as `alert(message)`, `prompt(message, default)`, and `confirm(question)`, but we can also create our own functions.

To create a function, we use a *function declaration*. The `function` keyword comes first, followed by the *function name*, a list of *parameters* inside parentheses separated by commas, and finally the *function body* enclosed in curly braces.

After creating a function, we can call it using:

```js id="8z7gk2"
functionName();
```

This executes the code inside the function.

Functions help avoid code duplication. If a change is needed later, the code only has to be modified in one place.

## Exercise 068

A function can also access variables declared outside of it and may even modify them.

An external variable is only used if there is no local variable with the same name.

If a variable with the same name is declared inside the function, the local variable *shadows* the external variable.

Variables declared outside functions are commonly called *global variables*. They are accessible from any function unless they are shadowed by local variables.

Try to minimize the use of global variables whenever possible. Most variables should exist within functions rather than in the global scope.

## Exercise 069

We can pass arbitrary data to functions through parameters.

When a function is called, the values provided are copied into the function's local variables, which are represented by its parameters.

As a result, passing a variable as an argument does not modify the original variable. The function receives its own local copy of the value.

When a value is passed to a function, it is called an *argument*.

A *parameter* is a variable listed inside the function declaration, while an *argument* is the actual value supplied when the function is called.

## Exercise 070

If a function is called without providing an argument, the corresponding parameter becomes `undefined`.

This is not considered an error. The function still executes, but it receives `undefined` as the parameter value.

To avoid this, we can define a *default parameter value* using `=` followed by the desired default value.

If no value is provided, or if the argument is explicitly `undefined`, the default value will be used instead.

Default values are not limited to simple strings or numbers. More complex expressions can also be used, and they are evaluated only when the parameter is missing.

Before default parameter syntax existed in JavaScript, developers often used explicit checks for `undefined` or relied on the `||` operator to assign default values.

## Exercise 072

A function can return a result using the `return` statement.

A `return` statement may appear anywhere inside a function. Once execution reaches it, the function immediately stops and returns the specified value.

A function may contain multiple `return` statements, each corresponding to a different execution path.

It is also possible to use `return` without a value. In this case, the function simply stops executing.

If a function does not explicitly return a value, it is treated as if it returned `undefined`.

Therefore:

```js id="z4frv7"
return;
```

is equivalent to:

```js id="h1n82w"
return undefined;
```

Be careful when returning long expressions.

JavaScript automatically inserts a semicolon after a standalone `return` statement. Because of this, writing the returned expression on the next line may produce unexpected results.

If the expression spans multiple lines, keep the opening parenthesis on the same line as `return` and place the remaining expression inside those parentheses.

## Exercise 073

Functions represent actions, so their names are usually verbs.

A function name should be concise, precise, and clearly describe what the function does.

It is common to start function names with a verbal prefix that loosely describes the action being performed:

* `show...` - display something
* `get...` - return a value
* `calc...` - calculate something
* `create...` - create something
* `check...` - verify something and return a boolean value

Examples include:

* `showMessage(..)`
* `getAge(..)`
* `calcSum(..)`
* `createForm(..)`
* `checkPermission(..)`

By looking at a function's name, we should be able to understand what kind of work it performs and what kind of value it returns.

A function should do exactly what its name suggests and nothing more.

Two independent actions usually require two separate functions, even if they are often used together. In such cases, a third function can be created to call both of them.

Examples of poor design:

* `getAge()` should not display an `alert`.
* `createForm()` should not modify the document.
* `checkPermission()` should not display an "Access granted" or "Access denied" message.

Some frequently used libraries employ extremely short function names. For example, *jQuery* uses `$`, and *Lodash* uses `_`.

These are exceptions. In general, function names should be descriptive and concise.

## Exercise 074

Functions should be short and perform exactly one task.

If a function becomes too complex, consider splitting it into smaller functions.

Smaller functions are easier to test, debug, reuse, and understand.

Their existence alone often makes the code more readable because each function can be given a meaningful name that explains its purpose.

In the example shown in the exercise, the function `isPrime(n)` is considered *self-descriptive* because its name clearly communicates what it does.

This style of code is often referred to as *self-documenting code*.
